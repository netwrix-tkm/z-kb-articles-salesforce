Summary: For troubleshooting purposes, it may become necessary to reconfigure a storage profile without knowledge of the back-end database server. This article describes one way to recover that information when the end user doesn't have the information.

Issue: Customer has a need to create storage profile from scratch without knowing their SQL Server or Instance names. This can become necessary as a result of a SA Vault Service reset (as happened in some versions of the upgrade from 7.2 to 8.2), since SA Vault Service removes the SPProfiles.XML file when configured.

Instructions: If StealthAUDIT is running with DEBUG level logging enabled, any recent job logs for jobs containing enabled a SQLScripting analysis tasks will record connection string information.
Steps:

  1. Locate a job in the tree with an analysis task that uses the SQL Scripting Analysis Module
  2. Open the Job Log
  3. Search for Data Source=
    1. Following the equal sign will be the SQL Server\Instance:port
  4. Search for Initial Catalog=
    1. Following the equal sign will be the database name
  5. For security there will be one of two entries
    1. Windows Authentication will appear as: Integrated Security=True;
    2. SQL Authentication will appear as:  Persist Security Info=True
    3. SQL Authentication will also show the username being used as well as asterisks representing the password
      1. User ID={YourUsername};Password=*********

 If the end user does not know the SQL Authentication password, they should now at least be able to escalate to someone who can help recover it.
