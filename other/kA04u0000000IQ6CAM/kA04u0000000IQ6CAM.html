Summary: How to handle a core hang type situation in StealthAUDIT.

Issue: If you're experiencing a core hang in StealthAUDIT, after ruling out as many factors as possible, if you absolutely need to submit an escalation for a core hang, you can follow these instructions.  

They may need to be adapted.  If you have suggestions or updates for this information, please let me know.  

Full instructions on how to take a process dump are located here:
https://na16.salesforce.com/articles/General/How-to-take-a-process-dump

Instructions: Start by ensuring that the customer is on

a) a supported version of StealthAUDIT (you should usually upgrade to the latest version and service pack for an issue like this)
b) the latest version of the data collector in question
c) if FSAA proxy is in use, verify the versions of the binaries (.dll, etc.) on the proxy match those in the SA installation directory


Troubleshooting Steps:

  1. Reboot the StealthAUDIT console
  2. Reboot the proxy server, if FSAA proxy is in use
  3. Disable any other schedules that may interfere.  We need to have only a single instance of SA running at this point. 
  4. Clear the core logs by deleting or moving this folder: C:\Program Files (x86)\STEALTHbits\StealthAUDIT\SADatabase\Logs)
  5. Set the host list to a single host (and 0 level scan, to reduce runtime, if you'd like)
  6. Run only the scan

If that resolves the issue, fantastic.  Add more hosts back in, or try the full host list.  Determine if a specific host is causing the problem.  

If all those conditions are satisfied, please provide: 

  1. procdump of StealthAUDIT while it is hung
    1. download procdump.exe from Microsoft, if you don't already have it
    2. from an admin command prompt, run: procdump -ma StealthAUDIT.exe or procdump -ma PID
  2. procdump of StealthAUDITRPC.exe on the proxy server
  3. applet and trace logs from the proxy
  4. job log
  5. core logs (C:\Program Files (x86)\STEALTHbits\StealthAUDIT\SADatabase\Logs)
