Intelligent Change Control

However, Change Tracker Gen 7 can operate in a proactive mode, learning to recognize change behaviors associated with approved Planned Change operations.

For example, were our ‚ÄòFIM_Test‚Äô change an approved patching operation that was going to be applied to all other hosts in the estate, instead of just reviewing and acknowledging the change when observed, we could instead create an Intelligent Planned Change based on the event.

This means that any other matching changes reported would also be automatically acknowledged, provided the changes were reported for approved devices, and were within the planned change time window.

In this way, ‚Äòknown-good‚Äô changes can automatically be categorized as Planned Changes, and conversely, any other changes not matching a Planned Change rule would be flagged as Unplanned Changes. Since the overwhelming majority of changes should be Planned, this greatly enhances the clarity of the breach detection capability of Change Tracker by reducing the ‚Äònoise‚Äô generated by regular, pre-approved changes.

Creating an Intelligent Planned Change is a straightforward ‚Äòpoint and click‚Äô operation.

Select the event or events to include in the Planned Change definition ‚Äì a separate rule will be created for each event.

As previously, use the Actions button but this time, select the ‚ÄòCreate a new Planned Change to capture these events‚Äô function.

The default Planned Change format is to include the ‚Äòevent donor‚Äô Device Group only, but other or different groups can be selected. Similarly, the Planned Change window start time is taken from the donor event time and left as an ‚Äòongoing‚Äô open ended window but these can also be edited.

In this example just click Apply.

Now delete the FIM_Test file created previously, then recreate the same file again.

Finally, create a further new file in /etc called ‚ÄòUnapproved_FIM_Test‚Äô then refresh the Events page.

Now events are being analyzed automatically, with the Intelligent Planned Change rules being applied to each one as it is recorded.

According to the default ruleset created, ‚ÄòFIM_Test‚Äô file operations are now approved so deletions, additions and even changes will all be automatically acknowledged as a ‚Äòknown-good‚Äô change.

However, other file change operations not matching the rule will still be flagged as Unplanned changes.

The default Planned Change rule created can be seen in more detail by clicking on the Planned link, then clicking on the ‚ÄòShow Rules‚Äô button. Planned Change rules can be edited and managed via the ‚ÄòSettings -> Planned Changes Administration‚Äô page.
