This document indicates our recommendations for monitoring the health of your NNT Server and Agents. We have listed several variables that can be monitored utilizing application performance monitoring tools for monitoring the Server and Agents. 

Services

WAS 
Track this on your NNT Hub Server to monitor the IIS Service


W3SVC
Track this on your NNT Hub Server to monitor the IIS Service


MongoDB
Track this on your NNT Hub Server to monitor that the Mongo Database is running


Gen7Agent
Track this on all of your Windows Agents & NNT Hub Server to monitor that the NNT Agent Service is running


nntgen7agent
Track this on all of your Red Hat and Debian based Distros to monitor that the NNT Agent Service is Running

Processes:

Gen7Agent.Service.exe
Track this on all of your windows based agents and hub server to make sure the NNT Agent process is running


w3wp or IIS Worker Process (w3wp.exe)
Track this on your NNT Hub Server to make sure that IIS Process is running


MongoD or MongoDB Database Server (mongod.exe)
Track this on your NNT Hub Server to make sure that the Mongo Database Process is running

.NET FRAMEWORK/Mono Runtime 3.5 - If you are using a NNT 6.5 Agent (Version Example: 1.0.0.106) - Make sure that this does not get uninstalled, or downgraded to an older version

.NET FRAMEWORK/Mono Runtime 4.5 - If you are using a NNT Gen7 Agent (Version 7.0.0.29) - Make sure that this does not get uninstalled, or downgraded to an older version. If you need to use an older version due to a critical application, use a 6.5 Agent instead.

.NET FRAMEWORK/Mono Runtime 4.6.1 - For the NNT Change Tracker Generation 7 Hub Server, (Version 7.0.1.48 or Newer) - Make sure that this does not get uninstalled, or downgraded to an older version.

Potential Syslog Thresholds / Messages:

Agent Error - This messaging type shows any errors that Change Tracker Agents are experiencing. It is reccomended that you build a threshold mechanism that searches for this kind of message from the change tracker server, and alerts the proper response teams.

Audit (Device Admin) - This messaging type shows any changes to device configurations. It is reccomended that you build a threshold mechanism that searches for this kind of message from the change tracker server, and alerts the proper response teams.

Audit (System Config Change) - This messaging type shows any changes to system settings for Change Tracker server configurations. It is reccomended that you build a threshold mechanism that searches for this kind of message from the change tracker server, and alerts the proper response teams.

Audit (User Administration) - This messaging type shows any changes to User Settings for Change Tracker console. It is reccomended that you build a threshold mechanism that searches for this kind of message from the change tracker server, and alerts the proper response teams.

Audit (Device Details Change) - This messaging type shows any changes to the details of an agent device. 

Device Offline - This messaging type shows when an agent goes offline. 

Device Online - This messaging type shows when an agent goes offline. 

New Device Registered - This messaging type shows when an new agent registers to the NNT Hub Server 

System Error - This messaging type shows when the Change Tracker Server is exeriencing an error. It is reccomended that you build a threshold mechanism that searches for this kind of message from the change tracker server, and alerts the proper response teams.

Planned and Unplanned Changes - This messaging type shows when the Change Tracker Server recieves planned or unplanned changes. It is reccomended that you build a threshold mechanism that searches for this kind of message from the change tracker server, and alerts the proper response teams.

Event Type: Communications - This messaging type shows communications between agent and hub server. It is reccomended that you build a threshold mechanism that searches for this kind of message from the change tracker server, and alerts the proper response teams.

Event Type: Audit - This messaging type shows audit information for Change Tracker, such as user account managment, configuration changes, etc. It is reccomended that you monitor for this, and build threshold mechanisims if needed.

Things to Keep in Mind

NNT Change Tracker Hub Server:

  • The percentage of memory in use on a server should not be monitored as MongoDB prefers to reserve as much as possible. The reserved memory is known as the "Commit Size" however, the memory that is actually in use is defined by the "Working Set". An Example of this is the screenshot below:

          

          

NNT Change Tracker Agents:

  • The NNT Change Tracker Gen7 Agent is built using .NET Framework. Applications that are based off .NET Framework may allocate different amounts of memory at different times. Keep in mind that .NET Framework has a Garbage Collecting function, which determines the best time for the system to clean up current memory banks which the application no longer needs. All in all, memory usage for .NET applications is a difficult metric to gauge, as in most scenarios, the application is not really utilizing all the memory that has been assigned to it, and the .NET Framework portion of the application will allocate and de-allocate the memory as per the system’s needs.
